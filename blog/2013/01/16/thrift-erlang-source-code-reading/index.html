Thrift Erlang实现源代码阅读

Jan 16th, 2013 | Comments

Thrift是一个可伸缩的跨语言RPC框架，它支持包括C++, Java, Python, Perl, Ruby, Erlang等多种编程语言。关于它的更多信息请参考Thrift官方网站，本文不是Thrift的入门指南，也不会解释Erlang这门语言的语法或OTP，而主要集中讨论Thrift里面的Erlang实现，假定目标读者了解Erlang，并且具有Thrift的使用经验，熟悉Thrift在其它语言的实现源代码更佳。

本文讨论的Thrift源代码版本为0.8.0，在官网可以下载。

虽然这里重点讨论的是内部实现的源代码，但是开始源代码之旅之前，简单回忆一下如何用Erlang编写Thrift客户端和服务器端是很帮助的。

1 编写客户端与服务器端

使用各种语言来编写Thrift客户端和服务器端的步骤都相同：在安装好thrift后，先编写好RPC的服务定义文件，再用Thrift的编译器编译生成某种语言的客户端和服务器端的RPC桩代码，然后再编写客户端和服务器代码。

Thrift 0.8.0版本的发布包中，tutorial目录下有自带的示例代码，就以它为例来说明。

1.1 编译发布包自带示例

可以先进入发布包根目录下的tutorial目录，这里有两个后缀为.thrift的RPC接口定义文件，不熟悉Thrift接口定义文件写法的读者，可以先看看官网的Thrift接口定义语言(IDL)文档，不熟悉Thrift使用步骤的读者也可以先读一下tutorial目录下面的README文件。使用命令

$ thrift -r --gen erl tutorial.thrift

调用thrift编译器生成对应的客户端和服务器端的Erlang桩代码，生成的新目录一般命名为gen-LANG，其中LANG即为上面命令中选项--gen后面的语言名字，刚才编译命令生成的新目录为gen-erl，此目录下生成的文件如下：

├── calculator_thrift.erl      -- 对应Calculator的接口代码
├── calculator_thrift.hrl
├── shared_constants.hrl
├── sharedService_thrift.erl   -- 对应SharedService的接口代码
├── sharedService_thrift.hrl
├── shared_types.erl
├── shared_types.hrl           -- shared.thrift文件中声明的所有类型
├── tutorial_constants.hrl     -- tutorial.thrift文件中声明的所有常数
├── tutorial_types.erl
└── tutorial_types.hrl         -- tutorial.thrift文件中声明的所有类型

其中生成的几种文件已经用注释标志出来，一种是接口代码，一种是类型代码。通过对比.thrift接口定义文件里面的内容和生成的Erlang代码，可以总结出各种IDL数据类型在编译后转化成的Erlang代码

IDL数据类型                   编译生成的Erlang代码
enum                        宏
基本类型常数                   宏
map                         dict
struct(optional选项不起作用)   record(Erlang里的record支持多层嵌套)
exception                   record
service                     针对每个接口生成function_info()函数，其中每个接口一组

我们重点看一下service的接口，以较复杂的service Calculator的calculate接口为例，它的IDL定义为

i32 calculate(1:i32 logid, 2:Work w) throws (1:InvalidOperation ouch),

对应的生成function_info()函数如下：

% calculate(This, Logid, W)
function_info('calculate', params_type) ->
{struct, [{1, i32}, {2, {struct, {'tutorial_types', 'work'}}}]};
function_info('calculate', reply_type) ->
i32;
function_info('calculate', exceptions) ->
{struct, [{1, {struct, {'tutorial_types', 'invalidOperation'}}}]};

可以看到function_info()函数的第一个参数是接口的名字字符串，第二个参数依次分别为此接口的参数，返回值和可能抛出的异常，对于参数和异常，都命名为struct的Erlang tuple，内部嵌套的参数列表中保留了参数索引，返回值类型则因接口定义是简单类型或复杂类型而异，IDL简单类型则直接映射为名字，复杂类型则同样转为命名为struct的tuple。为什么把接口的几个部分拆开用function_info()这样的函数调用来记录接口定义呢，了解Thrift其它语言(如C++, Python)实现的人看到这种形式会更觉得奇怪，在后面我们将会看到这种做法是与底层框架的调用紧密相关的，所以这里先跳过此疑问。

1.2 Thrift编译器

从.thrift接口定义文件是如何生成Erlang桩代码的呢？感兴趣的读者可以到发布包的compiler目录下阅读IDL解析器和各语言的生成器来一探究竟，这里限于篇幅就跳过这部分了。

1.3 示例客户端代码

看过编译生成的桩代码之后，我们再看一下0.8.0版本发布包里面对应的客户端和服务器端代码，在tutorial/erl目录下有：

├── client.erl        -- 客户端
├── client.sh
├── json_client.erl   -- json协议客户端
├── README
├── server.erl        -- 服务器端
└── server.sh

用注释标识出了客户端和服务器端的代码，其中json_client.erl与client.erl的不同只在于改用了json协议，这里就略过了。我们重点看一下客户端和服务器端的代码。

示例客户端的代码主要分成3个部分：

  1. 创建一个连接到指定服务器的客户端实体(这里的“实体”一词，是指向一个关联向某个内存对象的符号，本来想用“对象”一词，但Erlang是函数式编程语言)

     {ok, Client0} = thrift_client_util:new("127.0.0.1",
                                        Port,
                                        calculator_thrift,
                                        []),

    thrift_client_util:new()这个函数封装了创建并初始化客户端实体的代码，我们跟一下看看它做了哪些事情。

    Thrift的Erlang实现全部源代码都在lib/erl/src/目录，这个目录也就是本文重点讨论的源代码所在，打开目录下thrift_client_util.erl文件，会发现文件很短，没多少代码，其中导出的new()函数用于创建并初始化客户端实体：

     new(Host, Port, Service, Options)
   when is_integer(Port), is_atom(Service), is_list(Options) ->
     {ProtoOpts, TransOpts} = split_options(Options),

     {ok, TransportFactory} =
         thrift_socket_transport:new_transport_factory(Host, Port, TransOpts),

     {ok, ProtocolFactory} = thrift_binary_protocol:new_protocol_factory(
                               TransportFactory, ProtoOpts),

     {ok, Protocol} = ProtocolFactory(),

     thrift_client:new(Protocol, Service).

    其中split_options()用于分拣传入的选项，这个函数就定义在当前模块(也就是说在同一个文件)，其实现也很简单，就是用来划分开Transport和Protocol对应的选项。然后new()函数依次创建transport和protocol，最后创建并返回client，熟悉Thrift其它语言客户端写法的人会发现这段代码似曾相似，这几个创建客户端的步骤也很常见了，所以我们跳到下一步看看接口调用的代码。

  2. 使用上一步创建的客户端实体来调用RPC接口

    调用RPC接口的Erlang代码也只是常见的函数调用形式，没有什么出奇的地方，接口的参数以列表形式传入，返回值包含在tuple里面。由于Erlang中变量只能一次赋值，所以每次调用的返回值都会返回新操作过的客户端实体。下面是其中一个简单例子：

     {Client3, {ok, Sum1}} = thrift_client:call(Client2, add, [1, 4]),
 io:format("1+4=~p~n", [Sum1]),

    示例代码中也有带复杂参数的接口和抛出异常的接口的调用例子，也比较简单，这里就不一一展开了。

  3. 最后关闭与服务器的连接

    调用thrift_client:close()关闭连闭。

1.4 示例服务器端代码

示例服务器端的代码也较短，主要包含这两个部分：

  1. RPC接口的Erlang实现

    实现一个Thrift RPC接口，就是写一个名字参数返回值与IDL接口定义一致的函数，也可以直接throw声明的异常，这部分示例的函数都比较简单。

  2. 服务端器进程接口

    通过适当参数调用thrift_socket_server:start()启动进程，通过thrift_socket_server:stop()停止进程。

    最后handle_function()用于在当前模块分发接口处理函数，接口分为无回包（返回ok）和有回包（返回命名为reply的tuple）两种情况，这与IDL的接口定义，以及1的接口实现函数是对应的。

    特别值得一提的是，这种分发在Erlang这个动态语言中，从接口名（符号而非字符串）映射到接口的调用，只需要调用一下apply函数。同样的分发在C++中是这样做的：通过Thrift编译生成的桩代码，通过生成的processor类的process_fn()来查找接口名字与接口调用函数的映射，再回调到handler的对应接口实现函数。对比之下，静态语言中的从接口名到接口实现的分发比较纡回，代码量也较大，而动态语言的实现代码则简单清爽。

2 Erlang实现源代码

了解过Thrift客户端和服务器端之后，接下来看看Erlang的实现代码。按照官方文档给出的整体框架，Thrift自下到上可以分为4层：

+-------------------------------------------+
| Server                                    |  -- 服务器进程调度
| (single-threaded, event-driven etc)       |
+-------------------------------------------+
| Processor                                 |  -- RPC接口处理函数分发，IDL定义接口的实现将挂接到这里面
| (compiler generated)                      |
+-------------------------------------------+
| Protocol                                  |  -- 协议
| (JSON, compact etc)                       |
+-------------------------------------------+
| Transport                                 |  -- 网络传输
| (raw TCP, HTTP etc)                       |
+-------------------------------------------+

这个层次是根据代码的静态结构划分的，而非运行时的状态。在运行时下一层的接口由上一层调用，最后汇聚到顶层由顶层的Server启动服务，而并非每层都有对应的运行实体（线程或进程），从下层至上层的一层一层的回调。

在各种语言的实现中，每一层的接口都是设计好的固定接口。例如在C++中，transport这一层是由虚基类Transport衍生的单根类体系，根类Transport的open(), close(), read(), write(), flush()等成员函数确定了Transport层的接口，各种Transport的实现如TBufferedTransport, THttpTransport只是不同地实现了这些统一接口。

在Erlang中，跨模块（也就是跨文件）之间接口的一致性一般是通过behaviour来保证的，按同样的每层接口固定的实现思路，我们可以猜想在Erlang中每一层的接口都会写成behaviour。

下面我们就按这个分层结构，自下而上的逐层分析。

2.1 Transport

Transport层包括lib/erl/erc目录下文件名带transport字样的几个文件：

├── thrift_base64_transport.erl
├── thrift_buffered_transport.erl
├── thrift_disk_log_transport.erl
├── thrift_file_transport.erl
├── thrift_framed_transport.erl
├── thrift_http_transport.erl
├── thrift_memory_buffer.erl
├── thrift_socket_transport.erl
├── thrift_transport.erl
└── thrift_transport_state_test.erl

打开thrift_transport.erl，果然文件头就定义了Transport层的behaviour thrift_transport。

behaviour_info(callbacks) ->
    [{read, 2},
     {write, 2},
     {flush, 1},
     {close, 1}
    ].

整个Transport的代码比较简单，重点有两个，一是用来记录状态的record transport，定义为

-record(transport, {module, data}).

另一个是各个接口的实现方式，以read()函数为例

-spec read(#transport{}, non_neg_integer()) -> {#transport{}, {ok, binary()} | {error, _Reason}}.
read(Transport, Len) when is_integer(Len) ->
    Module = Transport#transport.module,
    {NewTransData, Result} = Module:read(Transport#transport.data, Len),
    {Transport#transport{data = NewTransData}, Result}.

这里只是简单调用具体实现模块的read()函数，在Erlang里面通过传递Module和function来进行跨模块的调用也是很常用的技巧了，在OTP里面经常见到。

下面我们看一个具体的Transport实现，这里选择较底层的thrift_socket_transport.erl来展开讨论，其它的Transport实现类似，限于篇幅就不一一展开了。

没有意外，thrift_socket_transport模块实现了上面定义的Transport behaviour thrift_transport，引入了另外的record来保存它独有的状态数据：

-record(data, {socket,
               recv_timeout=infinity}).
-type state() :: #data{}.

由于有Erlang有简单易用的gen_tcp模块，对应Transport层的代码也较为简洁，创建socket的函数是从transport工厂函数返回的：

new_transport_factory(Host, Port, Options) ->
    ParsedOpts = parse_factory_options(Options, #factory_opts{}),

    F = fun() ->
                SockOpts = [binary,
                            {packet, 0},
                            {active, false},
                            {nodelay, true} |
                            ParsedOpts#factory_opts.sockopts],
                case catch gen_tcp:connect(Host, Port, SockOpts,
                                           ParsedOpts#factory_opts.connect_timeout) of
                    {ok, Sock} ->
                        {ok, Transport} = thrift_socket_transport:new(Sock),
                        {ok, BufTransport} =
                            case ParsedOpts#factory_opts.framed of
                                true  -> thrift_framed_transport:new(Transport);
                                false -> thrift_buffered_transport:new(Transport)
                            end,
                        {ok, BufTransport};
                    Error  ->
                        Error
                end
        end,
    {ok, F}.

闭包返回的函数里面，先是创建到服务器的连接，然后根据传入的选项创建buffered_transport或framed_transport，这两者也是使用Thrift时常用的两种简单transport了，一种是流，一种带长度header。

有了创建好的socket，每次读写的时候就相当容易了，例如：

read(This = #data{socket=Socket, recv_timeout=Timeout}, Len)
  when is_integer(Len), Len >= 0 ->
    case gen_tcp:recv(Socket, Len, Timeout) of
        Err = {error, timeout} ->
            error_logger:info_msg("read timeout: peer conn ~p", [inet:peername(Socket)]),
            gen_tcp:close(Socket),
            {This, Err};
        Data ->
            {This, Data}
    end.

%% We can't really flush - everything is flushed when we write
flush(This) ->
    {This, ok}.

注意在外层包装的buffered_transport或framed_transport都有缓存数据，所以socket_transport的flush()函数为空。

这个socket_transport只用于客户端，服务器端由于需要按OTP的gen_server风格来写，所以放在另外的单独文件thrift_socket_server.erl中，它的实现代码将后面讨论。

2.2 Protocol

Protocol层包括lib/erl/erc目录下文件名带protocol字样的几个文件：

├── thrift_binary_protocol.erl
├── thrift_json_protocol.erl
├── thrift_protocol.erl

有了前面的经验，再读Protocol就变得轻车熟路了。通过阅读它的代码，我们知道thrift_protocol.erl定义了Protocol层接口的behaviour。以thrift_binary_protocol.erl为例来看具体Protocol实现，它做了IDL的数据类型的序列化值与Erlang atom的相互映射，同时针对各种IDL数据类型实现了对简单类型，结构类型，map, set, list 3种容器类型的读写（序列化与反序列化，或称做打包和解包协议）函数，Binary Protocol虽然设计简单，但由于IDL类型众多细节也比较多，了解Thrift实现的读者应该比较清楚，不清楚的话可以详细阅读源代码，这里就快速带过了。

2.3 Processor

Processor只有一个文件thrift_processor.erl。按Thrift的分层结构，Processor是挂接用户定制的IDL服务接口的地方，所以只存在于server。作为一个承上接下的层次，它的内部状态数据记录了IDL接口的实现模块service，接口分发模块handler和底层的protocol。

-record(thrift_processor, {handler, protocol, service}).

通过init()函数初始化Processor后server对应的连接处理进程进入的无限循环loop()函数，负责处理来自单个客户端的请求：

loop(State0 = #thrift_processor{protocol  = Proto0}) ->
    {Proto1, MessageBegin} = thrift_protocol:read(Proto0, message_begin),
    State1 = State0#thrift_processor{protocol = Proto1},
    case MessageBegin of
        #protocol_message_begin{name = Function,
                                type = ?tMessageType_CALL,
                                seqid = Seqid} ->
            {State2, ok} = handle_function(State1, list_to_atom(Function), Seqid),
            loop(State2);
        #protocol_message_begin{name = Function,
                                type = ?tMessageType_ONEWAY, 
                                seqid = Seqid} ->
            {State2, ok} = handle_function(State1, list_to_atom(Function), Seqid),
            loop(State2);
        {error, timeout} ->
            thrift_protocol:close_transport(Proto1),
            ok;
        {error, closed} ->
            %% error_logger:info_msg("Client disconnected~n"),
            thrift_protocol:close_transport(Proto1),
            exit(shutdown)
    end.

它从protocol中读出请求包头，并调用handle_function()进行实际的处理，代码中用了list_to_atom()来将从协议包中解出的函数名字符串转为Erlang符号，因为Erlang的符号在实现中不进行垃圾回收，所以如果有恶意的客户端发送百万级数格式正确但函数名不合法的请求，就可以把服务器拖垮，但将Thrift定位于内部服务使用，它的常用协议也没有加密机制，所以这里的代码可以认为是没问题的。

接下来重点看看实际请求的handle_function()，它调用handler_function()查询对应接口的所有参数及类型，这个函数是Thrift编译器生成的桩代码，在1.1节我们已经见过了，当前的疑问也得到了解答：桩代码之所以按文件thrift_service.erl定义的behaviour那样来生成，目的在于与这里的调用保持一致。然后根据得到的参数信息，从protocol中把请求包里所有参数解包出来，这个时候请求包的全部内容都已经被读取出来了，服务器端有了足够的信息来进行接口调用，于是调用接口的分发模块Handler的handle_function()函数来将请求分发到具体的接口实现函数，这个handle_function()函数我们之前在1.4节也已经见过了。

handle_function(State0=#thrift_processor{protocol = Proto0,
                                         handler = Handler,
                                         service = Service},
                Function,
                Seqid) ->
    InParams = Service:function_info(Function, params_type),

    {Proto1, {ok, Params}} = thrift_protocol:read(Proto0, InParams),
    State1 = State0#thrift_processor{protocol = Proto1},

    try
        Result = Handler:handle_function(Function, Params),
        %% {Micro, Result} = better_timer(Handler, handle_function, [Function, Params]),
        %% error_logger:info_msg("Processed ~p(~p) in ~.4fms~n",
        %%                       [Function, Params, Micro/1000.0]),
        handle_success(State1, Function, Result, Seqid)
    catch
        Type:Data when Type =:= throw orelse Type =:= error ->
            handle_function_catch(State1, Function, Type, Data, Seqid)
    end.

在handle_function()成功处理完请求后，它调用handle_success()查询桩代码中的对应接口返回值声明，在需要回包时进行回包，如果handle_function()抛出异常时，它调用handle_function_catch()查询IDL接口定义的异常声明，在需要回包时将异常进行适当的序列化后发送回包。发送回包的函数send_reply()则使用了下层protocol的write(), flush_transport()接口。

2.4 Server

作为这门语言的设计目标其中之一，Erlang对Server有很好的支持，提供了superviour, gen_server等OTP模块为Server长期稳定运行保驾护航，对OTP不熟悉的读者可以先读读由连城翻译的《Erlang/OTP并发编程实战》。

Thrift的Server也是一个定制的gen_server，其源代码包括lib/erl/erc目录下文件名带server字样的几个文件：

├── thrift_server.erl          -- 简单的单进程Server
├── thrift_socket_server.erl   -- 产品级多进程Server

其中单进程Server比较简单，在启动将socket设置成异步，在收到请求时通过start_processor()函数始化transport, protocol，并spawn一个processor开始处理请求。我们重点看一下多进程的Server代码。

多进程的socket_server可以通过start()函数来启动，这个我们在1.4节已经了解，它调用了start_sever()并通过gen_server的标准机制调用到init()函数。init()负责初始化Server并侦听服务端口，一旦有客户端连接进入，会调用new_acceptor()函数，后者会针对每个新连接创建一个新进程，并调用acceptor_loop()函数对应的来创建transport, protocol并调用processor的入口函数init()来开始处理请求，acceptor_loop()内执行的步骤与单进程的start_processor()函数类似，因为都是针对的都是单个客户端。其它的代码就比较简单了，这里略过。

2.5 Client

Client相关的源代码包括lib/erl/erc目录下文件名带client字样的几个文件：

├── thrift_client.erl
├── thrift_client_util.erl
├── thrift_reconnecting_client.erl

其中thrift_client_util.erl这个文件在1.3节我们已经讨论过。thrift_client.erl中是常用的简单客户端代码，我们将重点讨论它。thrift_reconnecting_client.erl是用gen_server实现的本地Server，伪装成Client，在简单客户端的基础上加上了自动重连服务器的功能，这里就不展开了。

thrift_client.erl中，常规的Thrift接口调用通过函数call()组织成为发出调用，接收回包两步，如代码所示：

-spec call(#tclient{}, atom(), list()) -> {#tclient{}, {ok, any()} | {error, any()}}.
call(Client = #tclient{}, Function, Args)
  when is_atom(Function), is_list(Args) ->
    case send_function_call(Client, Function, Args) of
        {Client1, ok} ->
            receive_function_result(Client1, Function);
        Else ->
            Else
    end.

如前文所述，Server端解包请求包时反序列化包头包体的逻辑，和打包回复包序列化包头包体的逻辑都放在processor中，Client端由于没有经过processor，于是这部分代码就放在Client模块中，Client打包发向Server的请求包和解包来自Server的回复包的代码分别在函数send_function_call()和receive_function_result()，后者根据回包的类型不同而执行不同的解包动作：如果是正常回包，则调用handle_reply()解之，如果回包是异常，则调用handle_application_exception()解之，整个代码还是比较简单的。

2.5 其它代码

目录lib/erl还有其它的一些测试代码，都比较简单，这里就不展开了：

├── include     -- 共用的hrl文件
├── src
└── test        -- 测试代码

3 总结

Thrift Erlang实现的所有主要代码到这里就讨论完了。总体来看，整个实现并不复杂，代码阅读起来也比较简明易懂，认真读起来很快就可以读完，对比C++或Python等其他语言的实现来看，Erlang实现的代码量相当小，这得益于Erlang对网络编程，服务器编写的优秀支持，以及函数式编程的清晰语义。
