当我谈机械键盘时，我谈些什么

Aug 25th, 2011

引子：天涯夕阳红，剑客论道时

大漠。
烈风如刀。

已到黄昏时分，火一般的红日缓缓落向西边，染红了西天的云彩，仿佛半个天空的云彩都被点燃了一般，流霞如幻，飞云炫目，好灿烂的一副光景。 风突然缓下来了。
飞舞中的黄沙也随之慢慢安静下来了，扬在空中的沙尘开始飘落，散于地上，那流动的黄色映照着红艳的夕阳，如丹沙，亦如鲜血！

黄沙落尽，沙地现出两个人影。
两人在距离不远的地方相对而立，其中一人白衣如雪，脸色如玉，两手垂立，一把长剑另在腰间，另一人青衣方帽，剑眉星目，两手束于身前，宽大的衣衫飘飘荡荡，却看不出有否带兵器。但这两个人都有一个共同点，那就是眼睛都紧紧盯着对方！

青衣人先开口了，“听说最近江湖上出现了一个人，每次出现都从东方而来，不知道你可认识？”
白衣人不露声色地说：“听说江湖上有一个人，发掘了道之源头，江湖上各门各派都由此而生，不知道你可认识？”
青衣人眼神闪动，似乎在仔细打量眼前的人，说：“想必你就是那个从东方来的人。”
白衣人说，“想必你就是那个发掘了道之源头的人。”
一阵微风吹过，带起两人的衣襟，青衣人双手藏在随风滚动宽大衣袖里，白衣人的白衣显得更加飘逸了。

青衣人问：“你为何而来？”
白衣人答：”为道而来。”
青衣人再问：”道在何处？”
白衣人答：“道在剑上。”
“剑可以用于防卫，也可以用于杀人。”
“据我所知，用于防卫的剑少，还是用于杀人的剑比较多。”
青衣人微笑，说：“听说你有一把好剑？”
白衣人抽出腰间的长剑，平托着向前半伸手臂，似乎是向青衣人展示自己的武器，然后说：”一流工艺，新颖设计，历经3代近10年的演进，才有了我手上这一把‘忍者’”。
在落日光线的照射下，剑上filco的标志闪烁着似灰似黑的金属光芒。
青衣人看到这样一把好剑，满面欣赏之色，不禁称赞道：“果然是一把好剑”。
白衣人将剑收回左手，问青衣人：“听说你也有一把非常厉害的剑？”
青衣人说：“的确，曾经我有一把很好的剑。”
“好在何处？在下愿闻其详。”
“诞生最早，专注品质，坚持30多年的用心，才能打造出那样的一把’g80’。”
白衣人眼睛一亮，说：“如此一把好武器，可否拿出来，让在下见识见识。”
“但那把剑现在已不在我手上了，我已不再用那样的武器。”
“对于一个剑客来说，能得到一把那样的剑是梦寐以求的事，我想像不到有什么原因会使你放弃它。”
青衣人展颜一笑，“因为我发现了比那把剑更厉害的，比所有的剑都更厉害的，世间最厉害的武器。”
白衣人愕然，喃喃道：“比所有的剑都更厉害的……世间最厉害的武器……”，思量沉吟半晌，然后眼神锁住青衣人，问：“那是什么？”
这时青衣人笑而不语，依然神秘地把双手收藏在宽大的衣袖中。突然，又一阵微风吹过，吹起了地上的细尘，吹动了两人的头发和衣衫，吹起了青衣人宽大的袖袍一角，而就在这电光火石的一刹那，青衣人迅速张开了那随风飘动的袖袍，向白衣人的方向扬去！
白衣人呆住了，没有动作没有反应，就在这一刹那，他已经看到了青衣人所说的，比所有的剑都更厉害的武器，世间最厉害的武器，就在青衣人那衣袖的一扬之中，他已经看到了，也已经领悟到了。

再谈机械键盘

上面仿古龙武侠体写了段两个机械键盘大品牌filco和cherry比较的小故事，应用中小学时候学到的40分钟作文大法中的一招：3段式作文，先写一个开头，在悬念中东扯西扯进入正文，谈谈机械键盘。

入手cherry g80黑青后，我发现用它来写代码手感真是越来越欢畅，虽然说几个卫星轴大键还是处于新键盘磨合期没用开，但青轴爽朗的打击感和强烈的段落感还是让手指迷恋不已。而在用了一段时间之后，发现原来的choc mini这款键盘的问题还是比较多的，举例其中比较蛋疼的几点：R4区键位排列比较密而且边框高，左上角的esc如同深陷边框中，这一排键盲打经常按错键，好在我是emacs党而不是vi党，不然不改esc键映射没法活，而且键帽印字很淡，看起来也很模糊。做工比较差。键帽既不完全是cherry的原厂高度，也不是OEM的高度，这样就不能换键帽，基本上失支了机械键盘的键帽可玩性，坏了键帽也只能找售后。后来把手上的choc mini黑茶出给了本地的一个二手键盘玩家，然后把cherry g80拿到了公司办公室替换原来的choc mini，周围的同事看到，纷纷来问，当听到价格之后，又纷纷表示没看出这么土的键盘有哪里贵的竟然值几百块。结果就是，从此以后这个并不安静的办公区里，每天多了一些敲打青轴键盘的“咔嗒”声。

上周在天下外设论坛上闲逛时看到帖子说，filco二代87白色青轴的国内行货在18号正式到货开卖，于是第一时间在国内的总代花尽旗下的键步如飞淘宝店下了单，在上周六20号下午，从顺丰快递拿到了这首批二代87青国内行货中的一把。开箱后整体看了一下，奶白带黄的材质比较细腻，有点像是瓷而不那么像塑料，键帽依旧小巧而涂层不那么明显，听说二代的涂层较一代有改进，手感更好而且不那么容易打油，不知道实际用起来情况如何。至于铭牌的话，87的铭牌放在方向键的下方，二代的铭牌是灰色的。其它的像键盘线，防尘罩那些与一代没有太多变化。

键皇到手，第一时间把键帽换了，换上从二嘟那里搞回来的全副淡彩pbt键帽。换键帽是个体力活，整个键盘的键帽全换更是一个体力活。花尽随键盘送了一个filco二代钢丝拔键器，用它来拔键虽然速度比较慢，但是它的力量传导比较好，拔起键来并不费劲，也不会伤到键帽。同时由于原来的键帽是abs，拔起来也很轻松顺利，对比拔cherry g80新键帽的时候大力一拔把卫星轴钢丝连带飞出的恐怖情景，换这个filco的键帽可谓是手到擒来，波澜不惊。最后换键帽完成，眼前得到一把filco二代87白青+87键淡彩pbt键帽的顶级奢华搭配，可以直追realforce。如此华丽的新搭档应该上图留念的，不过手上没有太好的拍照工具，暂时按下不表，日后再上图了。

filco这个品牌，其实它的配置性价比并不高，最为人诟病的一点就是abs键帽，无论是原厂的pom或者国产kbc/ducky的pbt或plu的pom都比它要好，不过综合来说，它的做工比国产品牌好，这两个的比较有点像apple notebook与神舟笔记本电脑做对比。而另一方面，它继承了日本产品精巧细致的特点，窄边框小面积的造型设计与原厂的传统大气走的是完全不同的路线。在国内用filco键盘的人关注它的人，还是挺多的，不过价格较贵是事实，当然市场价格受各种因素影响，这里面的水太深，简单的说，反正日本的电子产品来到中国卖就是贵，但中国人均收入远没有日本人高。

对比两个青轴键盘，原厂的键盘没有钢板，虽然说是青轴段落感很强，但触底的时候会感觉偏软，回弹也没有带钢板的有力，但是这种硬中带软的不会像薄膜键会觉得回弹无力手感枯涩，相反由于键帽底下有mx轴，回弹还是能感觉到弹簧的力道的，而filco的键按下回弹比原厂有力迅速，声音也相对响亮清脆一些，它的触底比较硬，所以使用起来感觉filco的键程比原厂的短，但实际上两种键盘用的都是同一种轴。

应用40分钟作文大法，后面一节续上开头古龙武侠体小故事的结尾，最后篇末点题，说说我对机械键盘，或者对所有工具的一种看法：工具是人四肢五官的延伸，工具的灵魂来自于人。所以工具是什么样的并不重要，重要的是人的本身，用这种工具去做什么样的事情，体现了什么样的价值和意义。

夕阳西下，天色渐暗，沙漠的夜晚很快就会降临。

人生的无奈与美好

Aug 22nd, 2011

今天约了海潮他们踢球，遇上一队身体素质和配合都不错的对手，被虐全场。踢完球离开华工，在校门前面的食街找地方一起吃饭，老地方韩国菜，记得这回是海潮毕业工作之后第一次请大家吃饭，也算是庆祝上月第一次出粮。席间说天论地，风生谈笑，吃完饭出来，一行人走在路上，当其时正在校门大石头附近，一旁的食街商铺灯光闪烁，不远处的校园沉默在夜色中，周杰说他很快就会离开广州去珠海，以后和我们各分两城，要抓住不多的机会一起再踢踢球。突然间似乎觉得有点感触，脑海中想起很多片断：大三足球比赛夺冠一起吃夜宵半夜走回宿舍，一群人喧哗而过的大学城大马路；毕业时各奔东西的匆忙；夏天午后深大的安静，热天夜晚深圳科技园整齐的道路和大冲拥挤的人流；最近一段时间相约在校园西区球场踢球的雨天和晴天。才发现，原来时光已经带我们不知不觉地走过这么多年，走了这么远。

周杰的想法，我们都知道已久了，与很多在毕业后依然在一起但分居两地的情侣一样，他与女朋友也一直面对着地理阻隔这个难题，到今天他终于做好准备动身去珠海与女友一起生活，也算圆了一个心愿。

虽然说最近一段时间也不时的约他们踢球，也一起吃饭，也经过这些地方，但平时大大咧咧后知后觉地活着，也没有意识到，原来这熟悉而平凡的一切，也有一天会失去会变残缺，只是在有人将离开的时候，才特别的觉得要珍惜。

回家的路上，夜色慢慢变深，缓缓前进的公交车外，广州城中的路灯相互交替，映射出灯光从远处渐渐拉近，接近而又渐渐远去，望着窗外的夜，思绪如野马乱奔。

在我这个局外人看来，从分居两地走到一起生活，对于男女双方来说无疑是一种幸福，但是否也是人生的一种无奈？在这个时代这个节骨眼上说，我们80后这个群体最深刻地感受到时代改变的影响，从小到大很多不好的事情都上我们赶上了，到如今高房价，生活压力大，大学毕业生出路少等等，这些常常在电视报纸，新闻报道中出现的字眼和问题，终于有一天会间接或直接地影响到我们每一个人的生活。广州房价高，也许不是每一个人都有钱买得起，也许即使有了钱，也并不是每个人都希望继续留在这个制度依然陈旧分配愈变不均压力愈变大的城市。作为一个个单个的社会个体，细小的个人面对庞大的社会机器，终究无力改变太多，只能无奈的接受。面对这个时代的无奈，今天周杰做出了他自己的一个中长期选择，而更多的80后朋友，他们如何做出他们各自的选择？

从个人的角度来看，一个男人为了另一个女人，放弃他已有的事业，到另外一个地方去重头开始一切，对于这个男人来说，获得的幸福算不算也带着一份牺牲？如果这个故事里的男主角换成了我，我会因为一个女人而付出这么多吗？伟大的爱情是不是本来没有包含太多的计较，只有一味的付出和给予？周杰的确是找到他的幸福了，而到今天我仍然没有遇见属于我的那个女主角，没有可以做如此付出的对象，其实是不是可以说是一种失败或者悲衰？人生因无奈而不完美，但又同时因为爱情而变美好，在这个世界活了四分之一个世纪，我没有去跑步，却发现人生是这样混着无奈和美好的一种滋味……

很多问题也许没有所谓的答案，正如人生，如果它算是一个问题的话。人一辈子来世上走这么一遭，也许答案与否什么的并不重要，埋入黄土的是不是答案也不重要，也许并不是每个人都能找到所谓答案，但至少每个人都经历过寻找的过程。

《Effective C++》读书笔记

Jul 24th, 2011

昨晚在网上看到一条新闻说“伞哥”离开网易，跟着链接到了“伞哥”的博客上看，惊叹不已，既佩服“伞哥”写博客之多，也敬仰他对lisp的热爱和工作生活中的真性情。牛人都是乐于分享，长于撰文的。很久没有在我的blog上发文了，贴一份之前的读书笔记吧。看来有空还是多码码字，人生才能向着剽悍的方向发展。

Item 1: View C++ as a federation of languages.

C++是一种多范型编程语言，它主要有４种子语言：
1) C  经典的C语言部分
2) 面向对象  类，继承，多态等
3) 模板  C++的泛型编程部分
4) STL  标准模板库
每个子语言都有自己的编程规则，当多种子语言混合的时候要注意这些规则的不同。

Item 2: Prefer consts, enums and inlines to #defines.

用编译器特性比用预处理器特性编程更好。
1) 用const，enum hack来定义常量，替代#defines
2) 用函数(无论inline与否)来定义操作，而不用#define宏
3) 预处理的#include/#ifdef/#ifndef指命依然很有用

Item 3: Use const whenever possible.

C++里关键字const用法有多种。其中两种用法：
1) 用const修饰(成员/非成员)函数返回值，防止返回值被意外修改，例如operator*()
2) const成员函数，可以被const对象调用。
const共有两种，bitwise const和logic const。const成员函数保证对象的bitwise const，logic const需要程序逻辑来保证。
关键字mutable让non-staic成员变量在const成员函数中去掉bitwise const的限制，使得它可以被修改。
3) 如果对应的成员函数const版本和non-const版本实现一致，可以通过在non-const版本的实现中调用const版本来避免代码重复。

Item 4: Make sure that objects are initialized before they’re used.

C++中全局的非静态对象(包括简单类型，用户自定义类型)不会被自动初始化。
总是初始化对象。学会使用构造函数初始化列表。针对不同编译单元中相互依赖的静态全局变量初始化顺序问题，可以通过将全局静态变量嵌入到全局静态函数来解决。

Item 5: Know what functions C++ silently writes and calls.

如果用户没有定义，编译器会隐式自动生成类的默认构造函数，析构函数，复制构造函数，赋值运算符。

Item 6: Explicitly disallow the use of compiler generated functions you do not want.

默认构造函数，析构函数，复制构造函数，赋值运算符这几个成员函数，如果用户定义了自己的版本，则编译器不会再生成对应的版本，即使是private或没有实现。
可以通过将复制构造函数和赋值运算符声明为private来禁止复制和赋值，使用boost库的noncopyable类是其中一种方法。 

#include <boost/utility.hpp>
class UserClass : private boost::noncopyable {
// the definition of UserClass
// blabla...
}

Item 7: Declare destructors virtual in polymorphic base classes.

多态的基类(有任何虚成员函数的类)应该有virtual的析构函数。
不是多态的基类不应该的析构函数不应该为virtual。
std::string和所有STL容器类析构函数都不为virtual，不适合作为多态基类。

Item 8: Prevent exceptions from leaving destructors.

析构函数不能抛出异常，如果析构函数执行的语句有可能抛出异常，要在析构函数内部处理完毕。
如果类的用户需要析构过程中某些动作的异常作出反应，则将这些动作放到一个普通成员函数。
在析构函数处理异常的几种选择：
1) 终止程序
2) 吃掉异常，不向上抛出

Item 9: Never call virtual functions during construction of destruction.

在基类的构造函数中调用虚函数只会在对象的基类部分调用基类版本对应的虚函数，因为继承类的部分还没初始化。
不要在构造函数或析构函数中调用虚函数。

Item 10: Have assignment operators return a reference to *this

作为一个惯例，让赋值操作符返回*this的引用，以支持连续赋值语句。

Item 11: Handler assignment to self in operator=.

在赋值操作符成员实现里处理自赋值的情况。通过安排好语句的顺序，copy-and-swap两种方法来保证实现的正确性和健壮性。

Item 12: Copy all parts of an object.

写复制构造函数要注意复制整个类的所有部分，包括每一个成员变量和它的基类部分。
复制构造函数和赋值运算符的实现往往有类似或相同的部分，多写一个私有的共用函数来避免代码重复。

Item 13: Use objects to manage resources.

用对象来管理资源的关键两点：
1) RAII，资源一申请到立即创建资源管理对象
2) 资源管理对象用它的析构函数来保证资源得到释放
已有的库中几种资源管理对象：
1) std::auto_ptr，独有的智能指针
2) tr1::shared_ptr/boost::shared_ptr，引用计数实现的多享智能指针
3) boost::scoped_arry和boot::shared_array，1)和2)的数组版本，释放资源调用delete []

Item 14: Think carefully about copying behavior in resource-managing classes.

RAII对象的复制行为可以实现为以下几种:
1) 不允许复制，如用#### Item 6的noncopyable类
2) 对底层资源用引用计数方式，tr1::shared_ptr支持定制deleter
3) 复制底层资源
4) 转移对底层资源的所有权

Item 15: Provide access to raw resources in resource managing classes.

RAII类应该提供成员函数来返回底层资源。对库层资源的访问有两种方式，显式的调用此成员函数，或通过RAII类提供的类型转换成员函数。

Item 16: Use the same form in corresponding uses of new and delete.

正确使用对应形式的new和delete操作符：
1) new <-> delete
2) new [] <-> delete []
否则会造成undefined行为。

Item 17: Store newed objects in smart pointers in standalone statements.

申请资源来生成RAII类应该是单独一个语句，不要将它嵌入到其它语句中，小心因为语句执行顺序的不确定而造成的资源泄漏。

Item 18: Make interfaces easy to use correctly and hard to use incorrectly.

好的接口应该是易于正确使用，难于错误使用的。设计接口时应该在心里保持这样一种认识。
促进正确使用的方法包括接口的一致性，保持与内置数据类型行为一致。
避免错误使用的方法包括创建新类型，限制在类型上的操作，限制对象的取值范围，以及消除客户使用时的资源管理责任。

Item 19: Treat class design as type design.

类设计就是类型设计。设计类里可以从以下几个方面来考虑：
1) 新类型的对象如何被创建和销毁
2) 对象的初始化和赋值有什么区别
3) 新类型对象如何passed by value
4) 新类型对象的合法取值
5) 新类型需要配合哪个继承体系
6) 新类型需要哪种类型转换
7) 新类型应该包含哪些操作
8 ) #### Item 6中的4个编译自动生成函数，哪个需要声明为private
9) 新类型的隐藏需求，如效率，异常安全性的保证
10) 新类型需要多大程度的通用，类，还是类模板
11) 真的需要一个新的类型吗，还是在已有的类上添点功能

Item 20: Prefer pass-by-reference-to-const to pass-by-value.

对于类对象来说，pass-by-value会调用复制构造函数生成一个函数内的临时对象，也会产生切片问题。
合适的情况下应该多用pass-by-reference-to-const，避免调用复制构造函数。
对于内置数据类型和STL迭代器，函数对象，还是保持pass-by-value。

Item 21: Don’t try to return a reference when you must return an object.

当选择返回对象还是返回对象的引用时，考虑哪一种能提供正确的行为。不要过分考虑调用复制构造函数的效率问题。

Item 22: Declare data members private.

将类接口声明为public，将成员变量声明为private。
从封装性来看，protected并不比public封装性好多少。大部分情况用public和private足矣。

Item 23: Prefer non-member non-friend functions to member functions.

在可以实现相同的功能时，优先用non-member non-friend function，而不用member function，来提高封装性，名空间文件的管理灵活性，和相关功能的扩展性。

Item 24: Declare non-member functions when type conversions should apply to all parameters.

如果要在某个函数的所有参数上做类型转换，将它实现为non-member function，最好是non-friend。

Item 25: Consider support for a non-throwing swap.

为你写的新类型提供一个不抛异常的swap成员函数，还有非成员的swap函数，特例化std::swap模板函数。

Item 26: Postpone variable definitions as long as possible.

尽可能延后变量定义出现的时机。这样可以增加程序的清晰性，改善效率。

Item 27: Minimize casting.

C形式的转型：
(T) expression
T(expression)
C++支持4种特有形式的转型：
1) const_cast，拿掉对象的const特性
2) dynamic_cast，安全向下转型，执行性能较差
3) reinterpret_cast，低层次的不可移植转型
4) static_cast，强制隐式转型，如non-const转为const，int转为double
优先采用新形式的转型，它们更专用，更易用识别。
尽量避免转型。

Item 28: Avoid returning “handles” to object internals.

避免返回对象内部的handle(包括引用，指针，迭代器)，const成员函数应该保证logic const，能增加类的封装性，避免产生悬挂的handle。

Item 29: Strive for exception-safe code.

异常安全的函数有3种层次的保证：
1) 基本保证，如异常抛出，对象内部状态前后一致，但程序的状态不可知。函数一般可以提供这种层次的保证
2) 强烈保证，如异常抛出，程序回到之前状态不变。往往可通过copy-and-swap来实现
3) 不抛出异常，决不抛出异常

Item 30: Understand the ins and outs of inlining.

只将inline用在频繁调用的小函数上，头文件的函数模板不必一定是inline的。用inline来优化性能之前，先测量函数的调用频繁程度，确定函数值得用inline手动优化。

Item 31: Minimize compilation dependencies between files.

只依赖于声明，不依赖于定义，可以使文件之间的编译依赖最小化，有两种方法达到这种效果，一是Handle class(pimpl idiom)，一是Interface class。

Item 32: Make sure public inheritance models “is-a”.

public继承塑造出”is-a”关系，每个子类对象都是一个父类对象。

Item 33: Avoid hiding inherited names.

子类的名字会隐藏基类内的名字，如果你的确需要这种特性，用using声明可以将基类的名字变为可见。

Item 34: Differenctiate between inheritance of interface and inheritance of implementation

public继承总是接口继承。
基类的纯虚函数意味着让子类只继承接口。
基类的非纯的虚函数意味着让子类继承接口和实现。
基类的非虚函数意味着让子类继承函数接口及一份唯一的实现。

Item 35: Consider alternatives to virtual functions.

基于虚函数的继承有几种替代方法：
1) NVI手法的Template Method模式
2) 通过函数指针实现的策略模式(Strategy Pattern)
3) 通过tr1::function模板实现的策略模式(Strategy Pattern)
4) 经典的策略模式(Strategy Pattern)

Item 36: Never redefine an inherited non-virtual function.

绝不重新定义继承得来的非虚函数。

Item 37: Never redefine a function’s inherited default parameter value.

虚函数是动态绑定的，参数默认值是静态绑定的，因此绝不要重新重义一个继承而来的默认参数值。

Item 38: Model “has-a” or “is-implemented-in-term-of” through composition.

组合与继承在类关系上表达两种完全不同的意思。
组合，有两种意思，在应用领域，意味着”has-a”(拥有)的关系，在实现领域，意味着”is-implemented-in-terms-of”(根据某物实现出)的关系。具体是哪一种意思，取决于使用场景。

Item 39: Use private inheritance judiciously.

Private继承意味着两个类有”is-implemented-in-terms-of”关系，通常来说组合更好一些。

Item 40: Use multiple inheritance judiciously.

严重慎用多重继承，大多数需要用多重继承的场景都可以找到一个不用重继承的解决方案。

Item 41: Understand implicit interfaces and compile time polymorphism.

类和模板都支持接口和多态。
对类来说，接口是显式的，基于函数签名，多态则通过虚函数发生于运行期。
对模板来说，接口是隐式的，基于有效表达式，多态是通过模板实例化和函数重载的定位发生于编译期。

Item 42: Understand the two meanings of typename.

在声明模板参数时，类型关键字class和typename是可互换的。
关键字typename还有另一个独特的作用：标识嵌套从属类型名字。例外情况是在继承蕨类声明列表和成员初始化列表中，这时不需要typename——C++是一门折腾的语言，读到这里你已经了解这么多了，珍惜生命，远离C++。

Item 43: Know how to access names in templatized base classes.

在模板子类中通过使用”this->”可以指明使用基模板类中的成员。

Item 44: Factor parameter-independent code out of templates.

让模板实例化尽可能的共用代码来重复代码，避免模板造成的代码膨胀。
因非类型模板参数造成的代码膨胀，往往可通过以函数参数或成员变量替换模板参数的办法来消除。

Item 45: Use member function templates to accept “all compatible types.”

用成员函数模板为所有兼容类型生成函数。
对于复制构造函数和赋值操作符，即使声明了成员函数模板，也要声明常规的这两个函数。

Item 46: Define non-member functions inside templates when type conversions are desired.

当我们编写一个类模板，而它提供的相关函数支持所有函数参数都需要隐式类型转换时，将那些函数定义为类模板内部的friend函数。

Item 47: Use traits classes for information about types.

traits类保存了类型的相关信息，在编译期间可用traits类来进行类型匹配。

Item 48: Be aware of template meta-programming.

模板元编程(TMP)是以C++模板写成的，执行于C++编译器内的程序，有其独特的编程方法。
TMP被证明是个turing-complete的，但目前还处于发展的初期阶段。非主流，可亵玩。

Item 49: Understand the behavior of the new-handler.

set_new_handler()允许用户指定一个函数new-handler，在内存分配失败时被回调。
new-handler应该做的事情可以是：
1) 让更多内存变成可用
2) 安装另一个new_handler
3) 卸掉new-handler
4) 抛出异常bad_alloc(或子类)
5) 直接终止程序，abort或exit

Item 50: Understand when it makes sense to replace nwe and delete.

有许多理由需要自定义new和delete运算符，包括改善性能，检测使用错误，收集使用统计数据等。

Item 51: 自定义new和delete运算符时要遵守惯例

new操作符是应该是这样的：有能力处理0字节申请，在无法申请到足够内存时，调用new-handler。 delete操作符也应该能处理null指针。

Item 52: Write placement delete if you write placement new.

placement new和placement delete是成双出现的，不能只写一个。定制placement new和placement delete小心遮掩它们的全局默认版本。C++陷阱很多的。

Item 53: Pay attention to compiler warnings.

写C++代码，一定要留意编译器警告，并熟悉这些警告的意思。不然就自求多福吧。

Item 54: Familiarize yourself with the standard library, including TR1.

C++98标准包括：标准模板库STL，io流系统，国际化支持，数学运算支持，异常体系，C89标准。
TR1标准覆盖范围比较广泛，不一一列举止，目前没有成熟的实现，boost可以作为一个临时候选。

Item 55: Familiarize yourself with Boost.

boost是一个社区，一个网站，一个开源C++库。
Boost为TR1的多个组件提供了不错的实现。
Boost是一个从飞机大炮到浴缸马桶无所不包的库，也是一个脑死亡的最佳去处，居家旅行必备。
