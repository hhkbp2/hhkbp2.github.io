刀丛小诗

Mar 9th, 2013

近日两会召开，从各种新闻报道看来，这个国家将进行又一次的领导层换届，在我工作的地方，类似的事情也在进行。在“云统治”的游戏背景设定下，这种事对于我等星斗市民并不会带来本质的变化，任它东边日出西边雨，只管一蓑任前行。从我个人的角度看，年初向领导提出加薪一事无果，长期问题还是没有得到解决，我对这份工作已没有什么期望。程序员要求提薪这件事情，马丁大叔在《程序员的职业素养》里提出了一个解读，比较有趣，很多时候事情的本质不仅是表面看起来那么回事。

早春三月，草长莺飞。每年的这个时间前后，都会发生有人离开有人加入的情景，就像在前领导人当政的时期。在人情淡如水的这个国家与时代，人们习惯了用“天下熙熙皆为利来，天下攘攘皆为利往”这样的说法来解释所有发生的事情，难道真的有那么一种普适的理论，能够解释这个世界所有的万事万物吗？在疑邻盗斧的人眼中，全世界只有偷了斧头的人。在纷繁多变的局势下，有的人主动退出乘风而飞，有的人成为炮灰黯然远去，也许我只是有点不太习惯离别来得太突然，想起来一首旧时的诗：

惯于长夜过春时，挈妇将雏鬓有丝。
梦里依稀慈母泪，城头变幻大王旗。
忍看朋辈成新鬼，怒向刀丛觅小诗。
吟罢低眉无写处，月光如水照缁衣。

这个鲁迅，在城头大王旗变来变去之际，还写这样的诗句，我想他除了胆子够大之外，心里一定有一种物伤其类的伤感，有一种难言的孤独。到来与离开，开始与结束，这里面的关联与区别，尤如镜子的两面出现在他的脑海中。

他想到每个人出现在这个世界以及这个地方的原因与意义，还有，那个他弃医从文走向的理想，是的，他清楚看到当时被腐蚀的社会那种僵化与衰败，他希望通过写一些东西，改变很多的人，改变一下这个世界——当然，它必须是好的改变，必须是往进步的方向。

他想到生命的价值，儒生在花花世界熟习经书伦理，佛陀眼望空茫的宇宙，灰鹤飞在以太一般无穷无尽的云海仙境，湿婆在山上跳着永恒的舞蹈。并非一切都注定有意义，正如人们不一定相信有命运。他所处的当前这个时代，不一定是最好的时代，历史有可能顺利向前，像在冲刷出来的三角洲上的河流，它也可能曲折跌荡，像海边悬崖的身形曲线。从大的时间尺度来看已知的历史，它的潮流多数是向前发展，对此他觉得很乐观，在这股潮流之中他只是一个小水滴，有没有这一个水滴潮流依旧是潮流，但水滴它在潮流中跳动着水滴的生命力，故成为水滴。

他想到去过的不同地方，见过的不同的人，有过的不同的谈话，做过的不同的事情，其实他从头到尾只做了一件事情，又或者说其实他根本没做任何事情。如今，他所有的，不过是这如水的月光，和一首小诗。于是人生中那些像台风一样不可逃避又难以捉摸的遭遇经历渐渐沉淀下来，玩骰子的上帝身影暂遁，过去发生的事情消逝，像是有了一个隔断电场的法拉第笼一样，这些问题不再困扰着他，不再有明枪暗箭，不再有反复动荡，不再有鸟尽弓藏。他渐渐洗去了外部的喧闹，剥除每一点多余的附着：虚荣，困窘，得意与恐惧，他体会着自然与时间在昼夜流转，渐渐安于淡泊和空灵。

最后他坐在书桌前，写下这首小诗。

无家可归的Java

Feb 23rd, 2013

Java千层糕

因为工作需要，近3个月我一直在研究Hadoop，读了HDFS以及HDFS RAID的不少Java代码。由于很久没用Java这门语言，有很多知识都忘记了，所以重新把书页已经发黄了的《Java编程思想》(第4版)找出来又翻了一遍。这本书是我毕业时带出学校的两本编程书之一（另一本是《The C++ Programming Language: Special Edition》），当年当作圣经来反复阅读一本编程书，几年后重读，很多东西感觉都不一样了，有种“横看成岭侧成峰”的陌生之感，还是学生的时候只知C++, Java，后来又自学了Python, Common Lisp, Erlang等编程语言，回过头来看看，发现Java其实也并不是曾经眼中那样的美好。

对于靠某门编程语言吃饭的程序员来说，将那门语言比喻作情人也毫不过分，因为说不好每天接触那门语言的时间比与现实生活中的情人相处的时间还要多。对于Java这个情人，也不必从哆嗦的语法，笨重低效的风格等各个方面具体列出她的缺点了，单是网上时不时出现的像标题《Java已死》这类文章，就像说明一些问题了。我个人则更喜欢用一种感性的方法来认识她，硅父教父Paul Graham曾经写过一个文章《JAVA’S Cover》(中文译文见这里)，用了类似“闻香识女人”的方法来预言Java，10多年后的今天读起来，感觉此文还是很有洞察力。文中说Java的老东家Sun商业前景不乐观，那时还是2001年，近年Sun已经被Oracle收购，Java之父去了Google，由于许可证的变化Sun的Java实现被踢出各开源Linux发行版之外，最近更是爆出多个严重的安全漏洞，就连美国政府也建议用户停用浏览器里的Java，这是相当罕见的。

其实一直以来，Java这个世界就习惯了打补丁的方式，从语言的设计(如容器和泛型)到修正安全漏洞各个方面。不断的发布安全更新，但漏洞也不断的出现，从底层虚拟机的实现上看，代码积越来越厚，补丁打得越来越多，也引入了越来越多的漏洞。如此像千层糕一样堆叠上去，终有一天Java会不堪重负而轰然倒塌。也许是时候重写Java了，但就和过去无数次打补丁的理由一样，Java只能选择兼容历史，不可能大改而影响市场上已有的程序，因为商业永远是第一位的，Oracle不可能像Python社区那样勇敢的抛弃历史的负担，于是Java这个故事发展下去就只能是个悲剧了。

且看2013年即将发布的Java 8，这个版本推出的对lambda的支持，被吹捧为Java带来新变革的重点，用起来依旧笨重，在Lisper的眼中看来甚至可能会觉得迂腐可笑。Java未死，还有很多已经存在或将会展开的应用，在Java虚拟机上积累下来的无数商业强度的库和代码也将长期存在，但从语言演化的层面上来看，这条路的终点也不远了。几年之前流行学习Java，但到今天，可能更好的做法是学习那些JVM语言，如Groovy, Scala, Clojure等。

没有JAVA_HOME

很久没有在Linux上装过Java了，有点生疏。sun-java6-jdk这个包已经不在Debian/Ubuntu的软件源里面了，我找了一圈才找到OpenJdk的openjdk-6-jdk这个包，装完包之后，照常例要设置环境变量，包括

JAVA_HOME=/usr/lib/jvm/default-java
export JAVA_HOME
CLASSPATH=.:$JAVA_HOME/lib:$JAVA_HOME/jre/lib
export CLASSPATH
PATH=$JAVA_HOME/bin:$PATH
export PATH

我把这几个语句写到一个独立脚本(后缀为.sh)放到/etc/profile.d目标里面，让Shell在启动的时候自动加载了。为什么不把配置放到/etc/environment文件里面呢？就像网上很多写Java安装的文章所说的那样？因为后者限制较多，它并非一个脚本，所以不支持变量的引用，只支持简单的赋值，用起来并不方便。更好的做法就是放到脚本里面，如只限个人使用就是.profile或.bashrc文件，如果是系统级别的就是/etc/profile.d目录或/etc/bash.bashrc文件。关于系统变量的详细内容可参考Ubuntu的Environment Variables文档。

搞好之后，突然顿悟似的想到，为什么还需要每个Java用户手工配置这个JAVA_HOME呢，为什么不可以没有它？于是联想到Python，它也有一个PYTHONPATH变量可以用来指定模块的搜索路径，但是在Python里面即使完全不知道此变量，程序员也可以活得很舒服。对比的来看，为什么不能在Java的安装包里面自动加上几个环境变量的配置？让用户更舒适一点，有很大的困难吗？问题虽小，但对比来看这背后其实反映的是两种不同的文化，就像ESR的《大教堂和集市》所说的那样：一边是笨重的大公司，另一边是活跃的社区；一边是官僚，另一边是黑客。

Thrift实践

Feb 10th, 2013

接触Thrift已经快两年了，实际使用和阅读源代码过程中也积累了一些经验，很多时候觉得应该梳理总结一点什么写下来，但这件事一直没有做成。这次趁着春节假期，尝试将之前留下来的几个片段组织成文，还有一些章节还未构思好或未写好，先用TODO标记占坑，以后再完善。

实践1. 将thrift看作应用的RPC胶水

流行的开源协议框架有这几种：google的Protocol Buffer，Apache(最早来自Facebook)的Thrift，还有Hadoop社区的Avro。这3种框架有着各不相同的设计目标和实现，其中Protocol Buffer设计为google内部的常用少数几种语言提供协议打包/解包(序列化/反序列化)，Avro旨在为Hadoop提供各种语言的访问接口和云计算的通讯协议标准，而Thrift则定义为Facebook内部服务的协议通讯框架，后两者除了带有协议打包/解包功能之外，都提供了较完整的RPC框架和较多的编程语言支持，但是两者的设计有较大的不同，具体反映在协议自描述性等实现细节上的种种差异。Protocol Buffer最先出现，然后是Thrift，Avro在Thrift广泛流行之后才被设计出来，从设计目标来看野心比Thrift大得多。

本文的重点在Thrift，所以集中笔墨讨论它。Thrift的RPC框架具有的如下几个特点：

  1. 定义清晰的IDL(接口定义语言)，支持丰富的数据类型和数据结构，具有强大的表达力
  2. 较全面的多语言支持，支持常用的编程语言如C++, Java, PHP, Python，还支持不那么常用的Erlang，Haskell等，方便使用各种语言，对接各种框架
  3. 支持的协议版本升级，可配合使用灵活的客户端/服务器升级策略
  4. 内部实现层次分明，结构清晰，代码量不大

如此使得它很适合作为业务应用的RPC胶水，把协议/RPC的实现和业务逻辑的表达粘合在一起，在响应快速开发和应对业务变更方面较有优势，同时这层胶水也不太厚，保有相当好的简单性和透明性，方便运营/根据各种具体需要做修改。

实践2. 使用合适的数据类型

2.1 当需要blob时，优先使用binary类型而不是string类型

thrift支持两种字符串类型： 1. string类型，类似编程语言中常用的字符串类型，或Mysql中的VARCHAR/TEXT字符串类型，用于表示某种语言(字符集)特定编码的字符串 2. binary类型，类似Mysql的BLOB类型，用于表示二进制串

虽然在部分Thrift支持的编程语言中，binary和string两个类型产生的具体数据类型是一致的，如C++。但在其它语言，这两个类型产生的具体数据类型差异较大，参考下表：

IDL Type\Lang   C++             Java                   Python
binary          std::string     java.lang.String       str
string          std::string     java.nio.ByteBuffer    str

在Java中原生支持unicode，字符类型Char设计为16位，字符的值对应Unicode的代码点(code point)[1]，而语言中的String也与编码方式相关，同一个String在不同的编码方式(如UTF-8或GBK)下可能表示不同的内容，这样一来就决定了String类型不适合用于存储带任意字节内容的二进制串。

从输出中可以看到，同一个字符在Python的客户端值与服务器端一样，C++客户端打印出来的值是服务器值的二进制补码值，但在Java客户端，字符值受到编码方式的影响，经过编码方式演绎后的字符值与原来的字符值并不一致。而且在几种常见的编码方式下都得不到原来的字符值。

假设有一个C++的服务器端，向Java的客户端传输从数据库读出来的BLOB串，而thrift协议里使用的是String类型，那么客户端得到的数据就可能与原来数据库中的blob有出入，数据是否一致取决于数据库blob里的字符取值，程序的行为有时正确有时不正确，这样就引入了一个隐晦的bug，浪费了调试的时间。

很多情况下对Thrift的使用哪种语言是不能预知的，比如，本来是用C++后来改成Java，这样之前一直正常工作的代码就“碰巧”都不能正常工作了，所以，选择Thrift数据类型时需要考虑周全尽量通用，当然这也需要实践经验，在需要blob时，优先使用binary类型而不是string类型。

实践3. Thrift里面的同步与异步

  1. 同步服务器，同步客户端，同步调用，无异步
    最常用的默认方式。

实践4. thrift内部框架一瞥

按照官方文档给出的整体框架，Thrift自下到上可以分为4层：

+-------------------------------------------+
| Server                                    |  -- 服务器进程调度
| (single-threaded, event-driven etc)       |
+-------------------------------------------+
| Processor                                 |  -- RPC接口处理函数分发，IDL定义接口的实现将挂接到这里面
| (compiler generated)                      |
+-------------------------------------------+
| Protocol                                  |  -- 协议
| (JSON, compact etc)                       |
+-------------------------------------------+
| Transport                                 |  -- 网络传输
| (raw TCP, HTTP etc)                       |
+-------------------------------------------+
