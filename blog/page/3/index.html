FSM

在计算机领域，FSM(有限状态机)是一个在自动机理论和程序设计实践中很常见的术语，简单来说，有限状态机表示的是系统根不同输入/不同条件在各个状态之间进行跳转的模型。可以通过图或表来描述有限状态机，这种图或表一般被称为状态图/状态转移图(State Chart)或状态转移表。因为图更加直观，本文统一使用状态图来描述有限状态机。

在状态图里面，一般用圆圈或方框来表示状态，用箭头来表示状态之间的跳转，箭头可以带上跳转需要的输入或条件，也可以带附带其它描述。一个从空白处引出，没有源状态的箭头则表示整个系统的启动，启动后进入的第一个状态可以称为开始状态，可以用双重圆圈特别标出。整个状态图就是一个有圆圈，箭头及描述的有向图形。下面是一个简单例子。

上图表示一个接受二进制输入(输入为0或者1)，计算输入包含奇数还是偶数个0的状态机。其中S1状态表示”偶数个0”，S2表示”奇数个0”。系统启动后，沿着图中最左边的箭头进入S1状态，此时没有读入任何输入（0个0）。S1圆圈上方带1的箭头表示如果输入是1，则跳转到S1，即保持原状态不变。如果输入是0，则跳转到S2。其它箭头也可以类似理解。当全部输入都处理完之后，只需看当前状态是S1还是S2即可得出结论：输入具有奇数个0还是偶数个0。

由于状态机可以将问题整体的分解成各个部分状态及跳转，直观地对系统进行建模，所以它不仅被用于理论研究过程当中，而且被广泛用于程序设计实践，在操作系统，网络协议栈，各种分布式应用中都可以见到它的身影。

程序设计中的FSM

由上面的表述我们得知，FSM是对系统的建模，是将问题/解决方案以一种条理化系统化的方式表达出来，映射到人的认知层面，而要在程序中表达FSM，也需要一定的建模工具，即用某种代码编写的方式(或称之为FSM模式)，将FSM以一种条理化系统化的方式映射到代码层面。在程序设计领域，到底有哪些常用的FSM实现方式呢？下面我们来做一个简单的回顾。

从简单到复杂，下面我们浏览一下常见的几种FSM实现模式[2]。

a. 嵌套if-else/switch模式

自从1960年第一个Lisp实现引入条件表达式以来，if-else/switch语句[3]已经成为每个程序员手头必备的工具，每当需要”根据不同条件进入不同分支”，就搬出它来组织代码，这与FSM里面”状态之间根据不同输入进行跳转”的概念有简单的对应关系，这就使得if-else/switch语句成为人们要表达FSM时最先选择的方式。

嵌套if-else/switch具有形式嵌套，代码集中化的特点，它只适合用来表达状态个数少，或者状态间跳转逻辑比较简单的FSM。嵌套意味着缩进层次的叠加，一个像图1那么简单的实现就需要缩进4层，如果状态间的逻辑变得复杂，所需要的缩进不断叠加，代码在水平方向上会发生膨胀；集中化意味着如果状态个数增多，输入变复杂，代码从垂直方向上会发生指数级别的膨胀。即使通过简化空分支，抽取逻辑到命名函数[5]等方法来”压缩”水平/垂直方向上的代码行数，依然无法从根本上解决膨胀问题，代码膨胀后造成可读性和可写性的急剧下降，例如某个状态里面负责正确设置20个相关变量，而下翻了几屏代码之后，下面的某个状态又用到上面20个变量里面其中的5个，整个代码像一锅粥一样粘糊在一起，变得难于理解和维护。

a. 状态表

另一个比较流行的模式是状态表模式。状态表模式是指将所有的状态和跳转逻辑规划成一个表格来表达FSM。仍然以图1为例子，系统中有两个状态S1和S2，不算自跳转，S1和S2之间只有两个跳转，我们用不同行来表示不同的状态，用不同的列来表示不同的输入，那么整个状态图可以组织成一张表格：

S1 DoSomething, S2 null
S2 DoSomething, S1 null

c. 状态模式

在OOP的设计模式[6]中，有一个状态模式可以用于表达状态机。状态模式基于OOP中的代理和多态。父类定义一系列通用的接口来处理输入事件，做为状态机的对外接口形态。每个包含具体逻辑的子类各表示状态机里面的一个状态，实现父类定义好的事件处理接口。然后定义一个指向具体子类对象的变量标记当前的状态，在一个上下文相关的环境中执行此变量对应的事件处理方法，来表达状态机。依然使用上述例子，用状态模式编写出的代码如下：

e. HSM

上述几种模式中，状态之间都是相互独立的，状态图没有重合的部分，整个状态机都是平坦的。然而实际上很多问题的状态机模型都不会是那么简单，有可能问题域本身就有状态嵌套的概念，有时为了重用大段的处理逻辑或代码，我们也需要支持嵌套的状态。这方面一个经典的例子就是图形应用程序的编写，通过图形应用程序的框架(如MFC, GTK, Qt)编写应用程序，程序员只需要注册少数感兴趣的事件响应，如点击某个按钮，大部分其它的事件响应都由默认框架处理，如程序的关闭。用状态机来建模，框架就是父状态，而应用程序就是子状态，子状态只需要处理它感兴趣的少数事件，大部分事件都由向上传递到框架这个父状态来处理，这两种系统之间有一个直观的类比关系，如下图所示：

这种事件向父层传递，子层继承了父类行为的结构，我们将其称为行为继承，以区别开OOP里面的类继承。并把这种包含嵌套状态的状态机称为HSM(hierarchical state machine)，层次状态机。

加上了对嵌套状态的支持之后，状态机的模型就可以变得任意复杂了，大大的扩大了状态机的适用场景和范围，如此一来用状态机对问题建模就好比用OOP对系统进行编程：识别出系统的状态及子状态，并将逻辑固化到状态及它们的跳转逻辑当中。

那么在状态机实现模式里如何支持嵌套状态呢？从整个状态图来看，状态/子状态所形成的这张大图本质上是一个单根的树结构，每个状态图有一个根结点top，每个状态是一个树结点，可以带任意多的子状态/子结点，每个子状态只有一个父结点，要表达嵌套状态，就是要构造出这样一棵树。

go-hsm

我用Golang编写了一个HSM框架go-hsm，设计要点如下：

  1. 用类来表示状态，局部化状态内部变量及逻辑，整棵状态树由具体应用构造
  2. 支持嵌套状态及行为继承，支持进入退出动作，
  3. 支持自定义事件，支持静态和动态两种跳转

它的代码在这里，go-hsm的使用例子则放在另一个项目go-hsm-examples。由于Golang本身的语言特点，有一些地方的实现较其它语言多了一些缺点，比如Golang里面的binding是静态的，为了将子类对象的指针传播到父类方法，要显式传递self指针，子类的接口函数也需要由应用重写。但由于HSM本身的灵活强大，go-hsm具有良好的可调整性及扩展性，是一个很好的状态机建模工具，一门灵活有效表达复杂状态机的DSL(Domain Specific Language)。

意兴阑珊

Oct 19th, 2014

时间过得很快，转眼就已经到了10月，也有一段比较长的时间没写博客了。人是一种适应性很强的动物，有时候随着周围环境的变化而改变了自身，但自己还不会意识到这种改变。村上春树在《世界尽头和冷酷仙境》里面说，时代的车轮一旦转动，只会越来越快，谁也无法让它停下来。而最冷酷无情的是，很多时候人们意识不到这种变得越来越快的节奏。

最近的我也是如此，由于工作上参与了一个新项目，最近几个月我一直保持忙碌，随着项目逐渐推进，节奏越变越快，我觉得要做的事情越来越多，时间越来越不够用。当夏天离开，秋风已然吹过，早晚的空气中都多少带点凉意的时候，我经历了今年少有的一次感冒，然后意识到也许是时候让自己的节奏慢下来了。

回头看看，我在最近几个月保持了每个月1万行左右的代码量，由于新项目的需要，我在github上写了一些库：

  1. GoVernor
    Go语言的工具链在使用之前需要手工设置一些环境变量如GOROOT, GOPATH，相比于其它很多编程语言在项目根目录添加配置文件来做项目管理的方式，我个人认为它是一种退步。每次切换项目目录都要改GOPATH，其实是件麻烦的事情，特别是对于带GUI的IDE，每次改环境变量都要重启，更麻烦的是windows/linux/mac改系统环境变量的方式各不相同。GoVernor这个命令行工具通过查找go语言项目的标准目录结构bin/pkg/src，来“猜测”正确的GOPATH值并在调用真正的Go工具链命令之前设置好GOPATH，在命令行环境中解决了“每次切换项目目录都要改GOPATH”这个问题
  2. rafted
    raft是一个在Paxos及它的各种变体流行10多年之后重新设计的一个分布式一致算法(协议)，它的目标除了达到等同于Paxos的分布式一致性之后，还特别强调算法本身的可理解性和工程上的易实现性。我们将要实现的分布式强一致bigtable模型存储系统选择了用raft来实现分布式一致性。而我负责这个部分的实现，在研读过市面上几种raft实现代码之后，我决定重新做一个全功能的支持可配置多种网络层的raft实现，结果就有了这个库，目前还有部分功能未实现完。
  3. rafted-starter
    rafted的使用例子。
  4. go-hsm
    将层次状态机(HSM)从C语言移植到Go，它是rafted的可理解性和正确性的基础。
  5. go-hsm-examples
    go-hsm库的使用例子，两个例子包含对应的状态转换图(state machine)设计和代码。
  6. go-logging
    开发rafted需要一个的日志库来帮助调试，它需要有支持为不同logger指定不同log level，不同输出文件，按指定log文件大小rotate和按指定log文件个数做backup，最好顺便支持logger hierarchy和常见的几种输出目的地，如syslog和支持通过配置文件配置，综合来看我意识到其实我需要一个与Python的logging模块几乎相同的Go语言日志库。在网上搜索过一轮之后，发现已有的Go语言日志库glog，log4go等都满足不了我的需求，最后只有自行实现一个，结果就写了这个库。 这个库基本上是logging从Python到Go的翻版，所有术语和用法都保持一致。因为时间所限，暂时还没写完所有handlers和配置文件相关的部分，所以暂时不能称之为当前最强Go日志库，但对从Python转到Go的人来说，上手成本基本上为零。
  7. go-strftime
    开发go-logging需要提供定制日志时间输出格式的功能，经过一轮折腾之后，我发现Go现有的用于定制时间格式的time.Format()函数并不能任意定制时间格式，远不如Python的strftime()函数那么灵活，于是为Go实现了一个strftime()。有趣的是，Go语言的设计者认为time.Format()要胜过strftime()。抛开现实世界，抛开实际的情况和需求不谈，任何人任何观点都可以是对的。

经过这一段时间之后，我突然发现，原来很多事情可以不是以现在的方式存在。肯承担干活多，有时也不一定是好事，没人认可终究也是白搭。有时候我会反思自己，为什么不是处女座的我会有一种处女座的心态，希望不断改良这个世界。世界终究是不完美的，无论从理论还是从实际的角度来看，为什么要执着于改进那些种种的缺憾之处呢？要改善要改良，就必须指出缺憾之处，推动一些事情，过程中往往得罪人或者吃力不讨好。最后也许达到也许达不到改良的效果，然而总在心里面落得个意兴阑珊。

系统性错误

Jul 27th, 2014

甲午战争

是为甲午年7月，距离1894年发生的中日战争已经120年，这个后来在中方被称之为甲午战争的事件，在同为甲午年的今年引起了很多的回顾和反思。甲午战争在1894年7月高升号事件正式拉开，到1895年4月中日代表签订《马关条约》结束，经过近9个月的战事之后结果清朝海陆两军大败，一个泱泱大国败于清朝当权者眼中的“蕞尔小邦”，从此滑落黑暗的深渊。

转眼之间，百年已过，清朝早已灭亡，拉开这么长的时间距离之后，再回顾甲午当年相关的史实，除了民族情绪上的唏嘘慨叹之外，更多是对客观事件的冷静反思和总结。甲午战争，在我看来是系统性错误的一个典型例子。

假设历史有如果的话，也许可以这样来描述为什么甲午战败是一个系统性错误：如果没有《天津会议专条》，也许中日就不会在朝鲜问题上不断增兵，终致酿成战争。如果没有洋务运动几十年耕耘，使得清廷在国力上产生自大的感觉，也许战争不爆发。如果慈禧和光绪的不是一方守旧强势，一方刻意求新这种关系的话，也许战争最后不会发动。如果清朝内部不是那样纷斗不休和战不定，也许对将到来的战争认识准备更充分一些。如果清朝海陆两军不是那么腐败废驰组织乏力，也许战争的结果不会是全面溃败……当一个清朝那样复杂庞大的系统，它的每个部分都出现问题的时候，就成为系统性错误，一些细小但不可避免的差错失误在它的每个部分相互传递和叠加，终于使之全面崩溃。

软件开发中的故事

相似的情景在软件开发中也是屡见不鲜，目前软件领域向着大规模、分布式的方向发展，一个复杂的软件系统往往由很多子服务、子模块组成。当一个系统内部各部分之间不能相互容错的话，就会造成错误在系统各处相互传递，由于分布式系统本身固有的大规模和复杂性，这种全局性的异常难以追踪发现，最后叠加放大造成严重结果，变成系统性的错误。

这方面我也有一些亲身经历，曾经有一次做一个系统，为了更好的系统健壮性，我提出在系统各模块内部都做容错，以模块边界的正确性作为屏障来避免系统性的错误，但由于各模块未能做好模块内的错误处理，结果由测试环境上线到实际生产环境之后，由于环境的变化单个模块出现错误，然后错误进行了跨模块的流动，形成系统性出错，造成业务上的故障。

如果对系统性错误没有任何认识，出现这种情况并不奇怪，但如果在对系统性错误已经有了经验并且有针对性设计之后，还是出现问题，就不能不引人深思了。最后我在这件事情上也收获了很多。

此处青山

一个多世纪之前，中国就因鸦片战争而进行洋务运动，几年后日本也在类似处境下开始明治维新，两个东亚国家都走过几十年的西化道路，然后甲午一战，一个走向半殖民地，一个得到飞速发展，结果悬殊。中体西用，不改国本只求器术，已经被历史证伪。一个多世纪的时间过去了，甲午战争的硝烟已随着时代远去，又一个甲午年的到来，提醒我们思考过去那个时代，以及，当今这个时代。

在附近某个公园中，有一个邓世昌衣冠冢，他是甲午海战中致远号的管带(船长)，大东沟一役他带领船上官兵英勇血战，最后壮烈殉国全船沉没于黄海，可能因为他是广东番禺人的缘故，后人在广州这里建了一个他的衣冠冢。我经常到公园跑步，也时不时经过那里，墓并不大，掩映在小径和树林的后面，只有一块不大的石碑记录他的事迹。

倘若在盛夏的7月黄昏，沿着石板小路走上山丘，入了树林之后，不久便到邓公的衣冠冢。四野无人，天色将暗，周围宁静的树林中响起虫鸟鸣声，鸣声仿佛穿越了120年的漫长岁月，像黄海的浪潮拍打海岸一样回荡于时空之中，邓公之魂就安息于此。

物换时移，而青山依旧。
