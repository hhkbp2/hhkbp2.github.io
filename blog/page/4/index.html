嵌套反引用有害

Mar 10th, 2013

本文翻译自《Nested Backquotes considered harmful》，只用于学习，请勿用于其它用途。

好吧，我想嵌套反引用（nested backquote）不是真的有害，因为你的确可以用它来做一些没有它就很难做到的事情；然而，它又的确会导致非常难以理解的代码（它对于混乱代码比赛[1]或许是有用的，但非常难以维护）。实际上，这篇文章的标题应该叫做“嵌套反引用令我头痛”，但我一直想写一篇标题叫做“那谁谁有害”的文章，于是，标题就是这个了。;-)

注：如果你不知道什么是嵌套反引用，它为什么有用，或者感兴趣于了解更多关于写出或理解这种典型地由嵌套反引用生成的，用于生成代码的代码，那么Alan Bawden的杰出论文《Quasiquotation in Lisp》值得一看（这是Jens Axel Søgaard所推荐的）。Alan的论文描述了这项技术的历史和概况，而且读起来很有趣。Paul Graham在他的书《On Lisp》的第16章也谈到了嵌套反引用，他说[2]： > 为了定义一个定义宏的宏，我们通常会要用到嵌套的反引用。嵌套反引用的难以理解是出了名的。尽管最终我们会对那些常见的情况了如指掌，但你不能指望随便挑一个反引用表达式，都能看一眼，就能立即说出它可以产生什么。这不能归罪于 Lisp。就像一个复杂的积分，没人能看一眼就得出积分的结果，但是我们不能因为这个就把问题归咎于积分的表示方法。道理是一样的。难点在于问题本身，而非表示问题的方法。

Bruno Haible最近在c.l.l.解释了避免嵌套反引用的两个不同方法。它们主要包含将嵌套反引用转换为简单反引用的做法。这两个方法他概述如下：

1. 在内层使用LIST, APPEND等函数代替反引用
2. 使用包含反引用的辅助函数

Bruno为每个方法提供了例子。

首先，下面给出的这段代码用了嵌套反引用：

(defmacro once-only (variables &rest body)
  (assert (every #'symbolp variables))
  (let ((temps nil))
    (dotimes (i (length variables)) (push (gensym) temps))
    `(if (every #'side-effect-free? (list .,variables))
       (progn .,body)
       `(let
          (,,@(mapcar #'(lambda (tmp var)
                          ``(,',tmp ,,var))
                      temps variables))
          ,(let ,(mapcar #'(lambda (var tmp) `(,var ',tmp))
                         variables temps)
             .,body)))))

原来的代码使用方法1转换后如下：

(defmacro once-only (variables &rest body)
  (assert (every #'symbolp variables))
  (let ((temps nil))
    (dotimes (i (length variables)) (push (gensym) temps))
    `(if (every #'side-effect-free? (list .,variables))
       (progn .,body)
       (list 'let
         (list ,@(mapcar #'(lambda (tmp var)
                             `(list ',tmp ,var))
                         temps variables))
         (let ,(mapcar #'(lambda (var tmp) `(,var ',tmp))
                       variables temps)
           .,body)))))

原来的代码使用方法2转换后如下：

(defun construct-binding (variable form)
  `(,variable ,form))

(defun construct-let-wrapper (bindings body-form)
  `(let ,bindings ,body-form))

(defmacro once-only (variables &rest body)
  (assert (every #'symbolp variables))
  (let ((temps nil))
    (dotimes (i (length variables)) (push (gensym) temps))
    `(if (every #'side-effect-free? (list .,variables))
       (progn .,body)
       (construct-let-wrapper
         (list ,@(mapcar #'(lambda (tmp var)
                             `(construct-binding ',tmp ,var))
                         temps variables))
         (let ,(mapcar #'(lambda (var tmp) `(,var ',tmp))
                       variables temps)
           .,body)))))

两个方法中的任一个，产生的代码的可读性都比原来代码好得多。

作为Bruno建议方法的一种替代，也许你会考虑使用Drew McDermott的BQ反引用工具（包含在YTools中）。Drew在YTools手册中对这个反引用问题有一个很好的解释：

反引用是Lisp的一个不可或缺的特性。然而在标准说明中它并非尽善尽美。我主要有两个不满：

  1. 当实现一个像反引用这样的机制时，需要3个东西：一个读取器，一个宏展开器，一个写入器。读取器将如`(foo ,x)的字符序列转换为如(backquote (foo (bq-comma x)))的内部形式（Allegro就是这样读取的）。之后宏展开器将backquote调用转变为像(list 'foo x)构造函数形式。写入器将(backquote (foo (bq-comma x)))输出为`(foo ,x)。
 不幸的是，Common Lisp标准没有详细说明宏（macro）是什么。因此，它依赖于具体实现。对比来看，普通的引用（quote）有一个定义良好的内部形式`(quote x)`，因此有一个定义良好的来自外部形式`'x`的转换。没有详细标准的问题就在于，要写一个你自己的工具配合读取器，宏取开器或写入器，是不可能的。例如，无法写一个可移植的代码遍历程序，针对反引用表达式来做一些特别的事情。事实上，一个Lisp实现甚至不要求有一个反引用的内部表示。读取器和宏展开器可以合并，以致于`` `(foo ,x)``可以读作`(list 'foo x)`。此外反引用写入器的行为并非良好定义的，因为无法区别一个列表构造形式是不是由反引用转化来的。

 为什么要与宏展开器交互呢，这里有个例子。你或许也想和读取器有交互。假设你希望创造一个泛化的反引用读取宏（就叫它`!@`吧），这个宏建造一些并非列表结构的东西，你可以将`(apply #'make-a-foo (list 'baz a) l)`简写作`!@(make-a-foo (baz ,a) ,@l)`。当这个表达式`!@(...)`被读入时，很多Lisp实现会报一个类似“Comma not inside a backquote”（逗号出现在非反引用表达式中）的错，没有可移植的办法来干预读取过程来使它变成合法的。
  1. 用于阐明嵌套反引用的规则是，逗号是与围绕它的最内层的反引用配对的。（同时提取它的参数到上下文中，以便下一个逗号匹配到下一个反引用，如此类推。）
 我想这是错误的，或者至少在某些情况下是错误的。我从左到右地读反引用代码，因此先看见最外层的反引用。人们会喜欢这样理解：从反引用的角度来看，所有在它里面的东西，除了用逗号标记的之外都是“不会动的（inert）”（即被引用住）。对于所有可能会出现在其中的表达式，除了反引用外，这是成立的。所以如果你正编辑一个复杂的反引用表达式：  

     '(foo (bazaroo '(fcn a ,x)))

 内层的引用不会“遮蔽”住`x`使它免于被求值。但如果将内层的引用改成反引用，那么遮蔽就正正要发生。你必须把它改成这样：

     '(foo (bazaroo `(,fcn a ,',x)))

 这个`,',`构造只是外观丑陋而已。它的唯一作用是将它的参数提取出最内层的反引用；你不能用`,,x`，因为这是说“当外层的反引用被展开时对`x`求值，得到`e`，然后当最内层反引用展开时求值`e`”。注意求值是次序是从外到内，然而嵌套反引用的规则是从内到外。非常，非常的令人迷惑。

这些不是巨大缺陷：99.9%的反引用不是嵌套的，而且几乎没人关心一个反引用的内部表示是什么。可是如果你感兴趣，文件bq.lisp提供一个可选的实现。

因此，正如Common Lisp里大多数的事物一样，如果你不喜欢某些东西，你可以改变它！当你需要写这类嵌套反引用的代码时，你有（至少）4种不同的可以考虑的选择（基于你有多少嵌套反引用工作要做，还有你个人对代码风格／可读性的偏好）：

1. 使用嵌套反引用
2. 在内层使用LIST, APPEND等函数代替反引用
3. 使用包含反引用的辅助函数
4. 使用一个反引用的替代实现

现在，出山吧，去写一些这样的代码：一种形式的代码不断的生成下一种形式，像生物进化的过程那样一代一代衍生。;-)

[1] 即Obfuscation Contest。国际C语言混乱代码大赛（IOCCC, The International Obfuscated C Code Contest）是一项著名的国际编程赛事。
[2] 此段译文引用田春（伞哥）等人翻译的《On Lisp中文版》对应章节。

刀丛小诗

Mar 9th, 2013

近日两会召开，从各种新闻报道看来，这个国家将进行又一次的领导层换届，在我工作的地方，类似的事情也在进行。在“云统治”的游戏背景设定下，这种事对于我等星斗市民并不会带来本质的变化，任它东边日出西边雨，只管一蓑任前行。从我个人的角度看，年初向领导提出加薪一事无果，长期问题还是没有得到解决，我对这份工作已没有什么期望。程序员要求提薪这件事情，马丁大叔在《程序员的职业素养》里提出了一个解读，比较有趣，很多时候事情的本质不仅是表面看起来那么回事。

早春三月，草长莺飞。每年的这个时间前后，都会发生有人离开有人加入的情景，就像在前领导人当政的时期。在人情淡如水的这个国家与时代，人们习惯了用“天下熙熙皆为利来，天下攘攘皆为利往”这样的说法来解释所有发生的事情，难道真的有那么一种普适的理论，能够解释这个世界所有的万事万物吗？在疑邻盗斧的人眼中，全世界只有偷了斧头的人。在纷繁多变的局势下，有的人主动退出乘风而飞，有的人成为炮灰黯然远去，也许我只是有点不太习惯离别来得太突然，想起来一首旧时的诗：

惯于长夜过春时，挈妇将雏鬓有丝。
梦里依稀慈母泪，城头变幻大王旗。
忍看朋辈成新鬼，怒向刀丛觅小诗。
吟罢低眉无写处，月光如水照缁衣。

这个鲁迅，在城头大王旗变来变去之际，还写这样的诗句，我想他除了胆子够大之外，心里一定有一种物伤其类的伤感，有一种难言的孤独。到来与离开，开始与结束，这里面的关联与区别，尤如镜子的两面出现在他的脑海中。

他想到每个人出现在这个世界以及这个地方的原因与意义，还有，那个他弃医从文走向的理想，是的，他清楚看到当时被腐蚀的社会那种僵化与衰败，他希望通过写一些东西，改变很多的人，改变一下这个世界——当然，它必须是好的改变，必须是往进步的方向。

他想到生命的价值，儒生在花花世界熟习经书伦理，佛陀眼望空茫的宇宙，灰鹤飞在以太一般无穷无尽的云海仙境，湿婆在山上跳着永恒的舞蹈。并非一切都注定有意义，正如人们不一定相信有命运。他所处的当前这个时代，不一定是最好的时代，历史有可能顺利向前，像在冲刷出来的三角洲上的河流，它也可能曲折跌荡，像海边悬崖的身形曲线。从大的时间尺度来看已知的历史，它的潮流多数是向前发展，对此他觉得很乐观，在这股潮流之中他只是一个小水滴，有没有这一个水滴潮流依旧是潮流，但水滴它在潮流中跳动着水滴的生命力，故成为水滴。

他想到去过的不同地方，见过的不同的人，有过的不同的谈话，做过的不同的事情，其实他从头到尾只做了一件事情，又或者说其实他根本没做任何事情。如今，他所有的，不过是这如水的月光，和一首小诗。于是人生中那些像台风一样不可逃避又难以捉摸的遭遇经历渐渐沉淀下来，玩骰子的上帝身影暂遁，过去发生的事情消逝，像是有了一个隔断电场的法拉第笼一样，这些问题不再困扰着他，不再有明枪暗箭，不再有反复动荡，不再有鸟尽弓藏。他渐渐洗去了外部的喧闹，剥除每一点多余的附着：虚荣，困窘，得意与恐惧，他体会着自然与时间在昼夜流转，渐渐安于淡泊和空灵。

最后他坐在书桌前，写下这首小诗。

无家可归的Java

Feb 23rd, 2013

Java千层糕

因为工作需要，近3个月我一直在研究Hadoop，读了HDFS以及HDFS RAID的不少Java代码。由于很久没用Java这门语言，有很多知识都忘记了，所以重新把书页已经发黄了的《Java编程思想》(第4版)找出来又翻了一遍。这本书是我毕业时带出学校的两本编程书之一（另一本是《The C++ Programming Language: Special Edition》），当年当作圣经来反复阅读一本编程书，几年后重读，很多东西感觉都不一样了，有种“横看成岭侧成峰”的陌生之感，还是学生的时候只知C++, Java，后来又自学了Python, Common Lisp, Erlang等编程语言，回过头来看看，发现Java其实也并不是曾经眼中那样的美好。

对于靠某门编程语言吃饭的程序员来说，将那门语言比喻作情人也毫不过分，因为说不好每天接触那门语言的时间比与现实生活中的情人相处的时间还要多。对于Java这个情人，也不必从哆嗦的语法，笨重低效的风格等各个方面具体列出她的缺点了，单是网上时不时出现的像标题《Java已死》这类文章，就像说明一些问题了。我个人则更喜欢用一种感性的方法来认识她，硅父教父Paul Graham曾经写过一个文章《JAVA’S Cover》(中文译文见这里)，用了类似“闻香识女人”的方法来预言Java，10多年后的今天读起来，感觉此文还是很有洞察力。文中说Java的老东家Sun商业前景不乐观，那时还是2001年，近年Sun已经被Oracle收购，Java之父去了Google，由于许可证的变化Sun的Java实现被踢出各开源Linux发行版之外，最近更是爆出多个严重的安全漏洞，就连美国政府也建议用户停用浏览器里的Java，这是相当罕见的。

其实一直以来，Java这个世界就习惯了打补丁的方式，从语言的设计(如容器和泛型)到修正安全漏洞各个方面。不断的发布安全更新，但漏洞也不断的出现，从底层虚拟机的实现上看，代码积越来越厚，补丁打得越来越多，也引入了越来越多的漏洞。如此像千层糕一样堆叠上去，终有一天Java会不堪重负而轰然倒塌。也许是时候重写Java了，但就和过去无数次打补丁的理由一样，Java只能选择兼容历史，不可能大改而影响市场上已有的程序，因为商业永远是第一位的，Oracle不可能像Python社区那样勇敢的抛弃历史的负担，于是Java这个故事发展下去就只能是个悲剧了。

且看2013年即将发布的Java 8，这个版本推出的对lambda的支持，被吹捧为Java带来新变革的重点，用起来依旧笨重，在Lisper的眼中看来甚至可能会觉得迂腐可笑。Java未死，还有很多已经存在或将会展开的应用，在Java虚拟机上积累下来的无数商业强度的库和代码也将长期存在，但从语言演化的层面上来看，这条路的终点也不远了。几年之前流行学习Java，但到今天，可能更好的做法是学习那些JVM语言，如Groovy, Scala, Clojure等。

没有JAVA_HOME

很久没有在Linux上装过Java了，有点生疏。sun-java6-jdk这个包已经不在Debian/Ubuntu的软件源里面了，我找了一圈才找到OpenJdk的openjdk-6-jdk这个包，装完包之后，照常例要设置环境变量，包括

JAVA_HOME=/usr/lib/jvm/default-java
export JAVA_HOME
CLASSPATH=.:$JAVA_HOME/lib:$JAVA_HOME/jre/lib
export CLASSPATH
PATH=$JAVA_HOME/bin:$PATH
export PATH

我把这几个语句写到一个独立脚本(后缀为.sh)放到/etc/profile.d目标里面，让Shell在启动的时候自动加载了。为什么不把配置放到/etc/environment文件里面呢？就像网上很多写Java安装的文章所说的那样？因为后者限制较多，它并非一个脚本，所以不支持变量的引用，只支持简单的赋值，用起来并不方便。更好的做法就是放到脚本里面，如只限个人使用就是.profile或.bashrc文件，如果是系统级别的就是/etc/profile.d目录或/etc/bash.bashrc文件。关于系统变量的详细内容可参考Ubuntu的Environment Variables文档。

搞好之后，突然顿悟似的想到，为什么还需要每个Java用户手工配置这个JAVA_HOME呢，为什么不可以没有它？于是联想到Python，它也有一个PYTHONPATH变量可以用来指定模块的搜索路径，但是在Python里面即使完全不知道此变量，程序员也可以活得很舒服。对比的来看，为什么不能在Java的安装包里面自动加上几个环境变量的配置？让用户更舒适一点，有很大的困难吗？问题虽小，但对比来看这背后其实反映的是两种不同的文化，就像ESR的《大教堂和集市》所说的那样：一边是笨重的大公司，另一边是活跃的社区；一边是官僚，另一边是黑客。
